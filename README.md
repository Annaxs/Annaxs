### `[07]` 🟣 BookKeeper Scale-Down — *Deep Production Operations*

<p>
  <img src="https://img.shields.io/badge/Apache%20BookKeeper-Scale%20Down-FF0055?style=flat-square&logo=apache&logoColor=white&labelColor=2B0057"/>
  <img src="https://img.shields.io/badge/10K%2B-TPS%20Live%20Traffic-00FFFF?style=flat-square&labelColor=2B0057"/>
  <img src="https://img.shields.io/badge/Zero%20Data%20Loss-Zero%20Downtime-FF0055?style=flat-square&labelColor=2B0057"/>
</p>

> Safely decommissioning BookKeeper nodes (bookies) from a live production cluster running **10,000+ TPS** — one of the most operationally sensitive tasks in Pulsar, requiring precise sequencing to avoid data loss or service disruption.

```
  THE CHALLENGE
  ─────────────────────────────────────────────────────────────────
  ✗ Bookies hold live ledger data — removing them risks data loss
  ✗ Under-replicated ledgers + cluster churn = potential message loss
  ✗ Live traffic (10K+ TPS) cannot pause during the operation
  ✗ No off-the-shelf runbook existed for our environment

  THE APPROACH
  ─────────────────────────────────────────────────────────────────

  Phase 1 — Pre-flight
  ├── Verify replication factor across all active ledgers
  ├── Confirm AutoRecovery daemon is active and healthy
  └── Ensure sufficient remaining bookie capacity post-scale-down

  Phase 2 — Controlled Decommission
  ├── Mark target bookie as read-only (stops new ledger assignments)
  ├── Monitor ledger migration: under-replicated → fully replicated
  ├── Wait for AutoRecovery to drain all ledger data off bookie
  └── Only proceed after: ledger count on bookie reaches ZERO

  Phase 3 — Safe Removal
  ├── Graceful shutdown of bookie process
  ├── Deregister from ZooKeeper/metadata store
  └── Validate cluster health metrics post-removal

  Phase 4 — Documentation
  └── Authored production-ready operational runbook
      (repeatable for future scale-down events)

  OUTCOME
  ─────────────────────────────────────────────────────────────────
  ✅ Zero data loss across all active ledgers
  ✅ Zero consumer-visible downtime
  ✅ Zero message redelivery events
  ✅ Production runbook published for future operations
```
